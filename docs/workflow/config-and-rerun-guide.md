# Configuration & Re-run Guide

What you can edit by hand, and what pipeline steps to re-run after.

> For the full pipeline walkthrough, see [simplified-workflow.md](simplified-workflow.md).
> For CLI tool options, see [cli-reference.md](../technical/cli-reference.md).

## Editable Config Files

These are the files you can safely edit by hand. Each entry shows what it controls and what to re-run after editing.

### 1. Book Rendering Config

**File:** `tools/lib/book-rendering-config.js`

Controls how `cnxml-render.js` renders each book's HTML. This is the main place where books differ from each other.

| Config Key | What It Controls | Example |
|------------|-----------------|---------|
| `noteTypeLabels` | Icelandic labels for note boxes (blue/green callouts) | `'link-to-learning': 'Tengill til náms'` |
| `titleTranslations` | Hardcoded phrase translations (Solution, Answer, etc.) | `'Check Your Learning': 'Prófaðu þekkingu þína'` |
| `endOfChapterSections` | Exercise types and end-of-chapter section structure | multiple-choice, short-answer, etc. |
| `excludedSectionClasses` | Which sections get their own standalone HTML pages | `['summary', 'exercises']` |
| `specialModules` | Modules needing special treatment | `m68859: 'periodic-table'` |

**After editing, re-run:**
```bash
# Re-render affected chapters (or all chapters in the book)
node tools/cnxml-render.js --book <book> --chapter <num>

# Then sync to vefur and regenerate TOC
cd ../namsbokasafn-vefur
node scripts/sync-content.js --source ../namsbokasafn-efni
node scripts/generate-toc.js
```

**Common edits:**
- Fix a misspelled Icelandic note label → edit `noteTypeLabels` → re-render
- Add a new note type for a book → add to `noteTypeLabels` → re-render
- Change how exercises are grouped → edit `endOfChapterSections` → re-render

### 2. Collection Order (Chapter/Module Structure)

**File:** `books/<book>/01-source/collection-order.json`

Maps chapters to their modules and provides English chapter titles. Auto-generated by `download-source.js` from OpenStax collection.xml.

**You should NOT edit this by hand** — it's generated from the source. If module ordering is wrong, the source collection.xml is the authority.

---

### 3. Chapter Status Files

**Files:** `books/<book>/chapters/ch<NN>/status.json`

Tracks pipeline progress per chapter (which stages are complete).

**Edit via CLI:**
```bash
npm run update-status <book> <chapter> <stage> <status>
npm run validate
```

**After editing:** No pipeline re-run needed — status files are metadata only.

---

### 4. Glossary / Terminology

**Files:** `books/<book>/glossary/*.csv`

Term glossaries used by `api-translate.js` to guide Miðeind's translation API.

**After editing:**
```bash
# Re-translate affected chapters (use --force to overwrite existing translations)
node tools/api-translate.js --book <book> --chapter <num> --force

# Then re-inject and re-render
node tools/cnxml-inject.js --book <book> --chapter <num>
node tools/cnxml-render.js --book <book> --chapter <num>
```

---

### 5. Faithful Translations (Reviewed Segments)

**Files:** `books/<book>/03-faithful-translation/ch<NN>/<moduleId>-segments.is.md`

Human-reviewed Icelandic translations, written by the segment editor's "Apply" function.

**You can edit these by hand** (they're just markdown with `<!-- SEG:... -->` markers), but the segment editor is easier.

**After editing:**
```bash
# Re-inject to produce updated CNXML
node tools/cnxml-inject.js --book <book> --chapter <num> --track faithful

# Re-render to produce updated HTML
node tools/cnxml-render.js --book <book> --chapter <num> --track faithful
```

---

### 6. Environment Variables

**File:** `.env` (project root, not committed)

| Variable | Purpose |
|----------|---------|
| `MALSTADUR_API_KEY` | Miðeind translation API key |
| `LOG_LEVEL` | Server log level (debug, info, warn, error) |
| `AZURE_*` | Microsoft Entra ID auth settings |
| `SESSION_SECRET` | Express session secret |

**After editing:** Restart the server (`npm run server:dev`). Pipeline CLI tools read `.env` automatically.

## Pipeline Dependency Matrix

When you change something, this tells you what to re-run. Read across the row.

| If you change... | Re-extract? | Re-translate? | Re-inject? | Re-render? | Sync to vefur? |
|------------------|:-----------:|:-------------:|:----------:|:----------:|:--------------:|
| Source CNXML (`01-source/`) | Yes | Yes | Yes | Yes | Yes |
| Rendering config (`book-rendering-config.js`) | — | — | — | **Yes** | Yes |
| Glossary terms (`glossary/*.csv`) | — | Yes (--force) | Yes | Yes | Yes |
| Reviewed segments (`03-faithful-translation/`) | — | — | Yes | Yes | Yes |
| MT output (`02-mt-output/`) | — | — | Yes | Yes | Yes |
| Chapter title translation (`chapter-metadata-segments.is.md`) | — | — | — | **Yes** | Yes |
| CSS in vefur (`static/styles/content.css`) | — | — | — | — | — (already there) |

**Key insight:** Changes flow left-to-right. A change at an early stage requires re-running all later stages. A change to rendering config only requires re-rendering — no re-extraction or re-injection needed.

## Quick Reference: Common Tasks

### "I fixed a translation in the segment editor"
The editor's "Apply" button writes to `03-faithful-translation/`. Then:
```bash
node tools/cnxml-inject.js --book <book> --chapter <num> --track faithful
node tools/cnxml-render.js --book <book> --chapter <num> --track faithful
```

### "I want to fix a note label (e.g., wrong Icelandic translation)"
Edit `tools/lib/book-rendering-config.js` → change the label → then:
```bash
# Re-render all chapters in the book (labels appear in every chapter)
for ch in $(seq 1 <last-chapter>); do
  node tools/cnxml-render.js --book <book> --chapter $ch
done
```

### "I want to add a new book to the pipeline"
1. Fetch source: `node tools/download-source.js --url <openstax-url> --slug <book-slug>`
2. Add config to `tools/lib/book-rendering-config.js` (copy an existing book, adjust labels/sections)
3. Add book to `server/config.js` `VALID_BOOKS` array
4. Extract a test chapter: `node tools/cnxml-extract.js --book <book-slug> --chapter 1`
5. Render a test chapter to catch any new patterns: `node tools/cnxml-render.js --book <book-slug> --chapter 1`
6. Review the HTML for any unstyled elements or missing labels

### "I want to re-translate a chapter from scratch"
```bash
node tools/api-translate.js --book <book> --chapter <num> --force
node tools/cnxml-inject.js --book <book> --chapter <num>
node tools/cnxml-render.js --book <book> --chapter <num>
```

### "I want to publish changes to the web"
```bash
cd ../namsbokasafn-vefur
node scripts/sync-content.js --source ../namsbokasafn-efni
node scripts/generate-toc.js
# Then commit and deploy
```

## Architecture: What's Generic vs. Book-Specific

The pipeline is ~85% generic. Book-specific variation is isolated in config.

```
GENERIC (same for all books)          BOOK-SPECIFIC (config only)
─────────────────────────────         ──────────────────────────────
cnxml-extract.js                      book-rendering-config.js
cnxml-inject.js                         ├── noteTypeLabels
cnxml-render.js (core logic)            ├── titleTranslations
api-translate.js                        ├── endOfChapterSections
chapter-modules.js                      ├── excludedSectionClasses
module-sections.js                      └── specialModules
cnxml-parser.js
cnxml-dom.js                          collection-order.json (per book)
                                      glossary/*.csv (per book)
```

**Adding a new book does NOT require code changes** — just a new config object (~30 lines) in `book-rendering-config.js`. The pipeline tools read the config and adapt automatically.
